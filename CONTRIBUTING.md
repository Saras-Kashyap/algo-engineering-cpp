# 🤝 Contributing to algo-engineering-cpp

First off, thank you for considering contributing to this repository. We are building the cleanest, most scalable LeetCode solution architecture on GitHub. 

To maintain our enterprise-grade standards, this project relies on a fully automated CI/CD pipeline, Test-Driven Development (TDD), and dynamic build systems. **Your focus should remain entirely on writing optimal C++ code and robust tests—our automation handles the rest.**

---

## 🛑 The Golden Rules of this Repository

Before you begin, please adhere to our strict automation boundaries:

> 1. **DO NOT edit `README.md` or `metadata/database.json`.** These are autonomously generated by our CI/CD pipeline upon successful merges. Manual edits will be overwritten.
> 2. **DO NOT modify `CMakeLists.txt` to include your tests.** Our build system dynamically discovers and links all valid test files automatically.
> 3. **DO NOT use `git add .`** Stage your files individually to ensure clean, atomic commits that match your commit messages and avoid leaking unrelated files into your pull request.

---

## 🛠️ The Standard Development Workflow

Please follow this standardized workflow to ensure your contributions pass our pipeline checks.

### 1. Fork, Clone, and Branch
Begin by forking the repository and creating an isolated branch for your solution.
```bash
git clone [https://github.com/Aditya-kumar-yadav/algo-engineering-cpp.git](https://github.com/Aditya-kumar-yadav/algo-engineering-cpp.git)
cd algo-engineering-cpp
git checkout -b feat/add-two-sum
```

### 2. Scaffold the Solution (Automated)
We enforce a strict boilerplate and directory structure. Use our Python automator to instantly generate your `.hpp` solution and `_test.cpp` files in the correct locations with the proper metadata.

Run the following command from the root directory:
```bash
python scripts/start.py --id [PROBLEM_ID] --topic [CATEGORY] --title "[PROBLEM TITLE]" --diff [Easy/Medium/Hard]
```

*Example Execution:*
```bash
python scripts/start.py --id 20 --topic stacks --title "Valid Parentheses" --diff Easy
```

### 3. Implement and Test (TDD)
1. **The Logic:** Navigate to the generated `.hpp` file in the `src/` directory and implement your optimal solution.
2. **The Validation:** Navigate to the corresponding `_test.cpp` file in the `tests/` directory and write your **Google Test (GTest)** assertions. 
   * *Code submitted without comprehensive tests covering edge cases will not be merged.*

### 4. Pre-Commit Validation (Required)
Before opening a Pull Request, you must verify that your code compiles and passes all tests locally. This ensures our cloud CI/CD pipeline remains green.

```bash
# 1. Configure and build the project
cmake -B build -S .
cmake --build build

# 2. Execute the test suite
cd build && ctest --output-on-failure

# 3. Verify metadata extraction (Optional but recommended)
cd ..
python scripts/auto_discover.py
```

### 5. Git Commit Hygiene
We strictly enforce **Conventional Commits**. Grouping unrelated files into a single commit is not allowed. 

Stage your files explicitly:
```bash
git status
git add src/topic_name/0000_problem.hpp
git add tests/topic_name/0000_problem_test.cpp
git commit -m "feat(topic_name): add optimal O(n) solution for Problem Name"
```

### 6. Submit Your Pull Request
Once your local tests pass and your commits are clean:
1. Push the branch to your forked repository.
2. Open a Pull Request against our `main` branch. 

Our GitHub Actions pipeline will automatically compile your code, execute the test suite, and upon a successful merge, autonomously update the central database and Markdown documentation.

## ⚠️ Manual Creation Fallback (Not Recommended)

If you are unable to use the Python scaffolding script, you must manually create your files with exact precision. **Failure to adhere to these naming conventions will cause the dynamic CMake build to ignore your files.**

1. **Source File (`src/`):** * Format: `[ID]_[problem_name].hpp` (e.g., `0001_two_sum.hpp`).
   * It **must** include this exact metadata header block at the top of the file:
     ```cpp
     /* * ID: 1 
      * Title: Two Sum 
      * Difficulty: Easy 
      */
     class Solution {
     public:
         // Implementation here
     };
     ```
2. **Test File (`tests/`):**
   * Format: You **must** append `_test.cpp` to the filename (e.g., `two_sum_test.cpp`).
